# Exchange Online 移行プロジェクト 要件定義書（案）

## 概要

本書は、○○株式会社様におけるメールシステムのExchange Online移行に関する要件を定義する。

---

## 1. プロジェクト概要

### 1.1 プロジェクト名

```
○○株式会社 メールシステム Exchange Online 移行プロジェクト（パイロット）
```

### 1.2 目的・背景

| 項目 | 内容 |
|---|---|
| 背景 | 現行のLinuxベースメールシステム（Postfix/Courier IMAP）の運用負荷、クライアント（Thunderbird/POP）の利便性制約、ID管理の煩雑さを解消したい |
| 目的 | 1ドメインを先行してExchange Onlineに移行し、Outlookによる利便性向上とMicrosoft 365基盤への統合を実現する |
| 期待効果 | クラウドメールボックスによる端末非依存化、GAL活用による連絡先管理簡素化、転送依存の解消、運用負荷軽減 |

### 1.3 プロジェクト期間

| 項目 | 日程 |
|---|---|
| プロジェクト開始 | 2026年5月（予定） |
| パイロット移行完了 | 2026年6月（予定・2ヶ月間） |
| ユーザーテスト期間 | 2026年7月〜9月（予定） |
| 本格移行判断（稟議） | 2026年10月 |

## 2. 現行環境

### 2.1 メールシステム構成

| コンポーネント | 製品/サービス | 配置 | 備考 |
|---|---|---|---|
| SMTPハブ | Postfix | AWS | ルーティング・送信 |
| メールボックス | Courier IMAP | AWS | 40ドメイン、ユーザーはPOP受信 |
| セキュリティ（受信） | FireEye | クラウド | MXがここを向く |
| セキュリティ（送信） | GuardianWall | AWS（EC2） | 添付ファイルURL化 |
| SMTP中継（メイン） | Postfix | AWS DMZ | 内部/外部の振り分け |
| SMTP中継（フォールバック） | Postfix | オンプレ内部DMZ | ほぼ未使用 |
| メールクライアント | Thunderbird | Windows | POP受信 |
| Webメール | Roundcube | AWS | ほぼ未使用 |

### 2.2 ドメイン一覧

| # | ドメイン | 用途 | ユーザー数 | 備考 |
|---|---|---|---|---|
| 1〜40 | （40ドメイン） | 部署・子会社等 | 要確認 | 今回は1ドメインのみ移行対象 |

※ 詳細なドメイン一覧はお客様より別途提供

### 2.3 メールフロー概要

```
【送信フロー】
Thunderbird / システムメール → Postfix（SMTPハブ）→
  ├─ 内部宛 → Courier IMAP
  └─ 外部宛 → GuardianWall → AWS DMZ SMTP → インターネット

【受信フロー】
インターネット → FireEye（MX）→ AWS DMZ SMTP → Postfix → Courier IMAP
                                └─ フォールバック → 内部DMZ SMTP → Postfix → Courier IMAP
```

### 2.4 ID管理

| 項目 | 内容 |
|---|---|
| ID管理システム | 電話帳/メールアドレス管理システム（AWS上、PHP + PostgreSQL）→ 統合アカウントシステム（オンプレ、LDAP） |
| AD連携 | あり（統合アカウントシステム → Active Directory） |
| Entra ID連携 | あり（Entra ID Connect構成済み、ユーザー＋セキュリティグループ同期） |
| メールアドレス作成フロー | ユーザー申請 → 電話帳システム → 統合アカウントシステム → AD（mail属性はほぼ空） |
| 特記事項 | 命名規則が緩く自由度が高い、1ユーザー複数アドレスの可能性あり |

### 2.5 現行環境の特徴

| 項目 | 現状 | 影響 |
|---|---|---|
| POP受信 | メールがローカル端末に保存 | サーバーにメール残らない、端末依存 |
| メール転送 | 横行している | セキュリティ懸念、お客様は防ぎたい意向 |
| AD mail属性 | ほぼ空 | 全件投入が必要 |
| 1ユーザー複数アドレス | あり | 設計時に要整理 |
| 共有アドレス | あり（詳細不明） | 実態確認が必要 |
| EXO環境 | 存在するが未使用、EWSのみオン | 紛れ確認が必要 |

## 3. 移行後の目標状態（To-Be）

### 3.1 システム構成

| コンポーネント | 製品/サービス | 備考 |
|---|---|---|
| メールボックス | **Exchange Online** | 移行対象ドメインのユーザー |
| メールクライアント | **Outlook** | |
| セキュリティ（受信） | FireEye | **継続使用、設定変更なし** |
| セキュリティ（送信） | **GuardianWall Cloud** | 添付URL化継続 |
| ID管理 | Active Directory + Entra ID Connect | **スキーマ拡張してメール属性投入** |

### 3.2 メールフロー概要（パイロットドメイン）

```
【送信フロー】
Outlook → Exchange Online → GuardianWall Cloud → インターネット

【受信フロー】
インターネット → FireEye → AWS DMZ SMTP → Exchange Online
                                          ※transport設定で移行対象ドメインをEXOへ振り分け

【未移行ユーザー宛の救済】
Exchange Online（Internal Relay）→ 内部DMZ SMTP → Courier IMAP
```

### 3.3 設計方針

| # | 方針 | 理由 |
|---|---|---|
| 1 | MX変更なし（DNS変更なし） | 影響範囲最小化、FireEye継続利用 |
| 2 | AWS DMZ SMTPのtransport設定で制御 | ドメイン単位の柔軟な切り替え |
| 3 | 未移行ユーザーはInternal Relay + 内部DMZ SMTP経由でCourierへ | 段階的移行中のメールロス防止 |
| 4 | 内部宛メールは外部セキュリティ経路に出さない | 不要なGWC経由を回避 |

### 6.3 ID運用整理のスコープ

ID運用整理（メールアドレスの命名規則、ライフサイクル管理等）は本PJのスコープに含まれるが、**導入フェーズと運用フェーズで役割分担**する。

#### 導入PJ（本PJ）での実施内容

| # | 項目 | 内容 | 成果物 |
|---|---|---|---|
| 1 | **現状把握** | 現行のID作成・変更・削除フローの調査 | 現状整理ドキュメント |
| 2 | **課題抽出** | 命名規則の課題、運用上の問題点の洗い出し | 課題一覧 |
| 3 | **運用設計（方針）** | EXO移行後のID運用方針の策定 | 運用設計書（方針編） |
| 4 | **移行時のルール** | パイロット移行時のメールアドレス投入ルール | 移行ルール |

#### 運用フェーズ（運用担当と協働）での実施内容

| # | 項目 | 内容 | 備考 |
|---|---|---|---|
| 1 | 命名規則の策定・適用 | 新規ID作成時の命名規則確定 | 運用担当と検討 |
| 2 | 承認フローの整備 | ID作成・変更の申請・承認フロー | 運用担当と検討 |
| 3 | ライフサイクル管理 | Joiner/Mover/Leaver各フェーズの運用ルール | 運用担当と検討 |
| 4 | 運用ドキュメント整備 | 運用手順書、FAQ等 | 運用担当が主体 |

#### ID運用整理の検討観点（JMLライフサイクル）

| フェーズ | 検討事項 | 現状（要確認） |
|---|---|---|
| **Joiner（作成）** | いつ・誰が・どのように申請するか | 電話帳システムで自由申請 |
| | 命名規則（firstname.lastname@等） | 緩い、自由度が高い |
| | 承認フロー | 要確認 |
| | 複数アドレスの付与ルール | 要確認 |
| **Mover（変更）** | 部署異動時のアドレス変更 | 要確認 |
| | 役職変更時の対応 | 要確認 |
| | エイリアス追加のルール | 要確認 |
| **Leaver（削除）** | 退職時のメールボックス対応 | 要確認 |
| | データ保持期間 | 要確認 |
| | 転送設定の扱い | 要確認 |

#### 導入PJと運用の境界

```
【導入PJ】                          【運用フェーズ】
                                    
現状把握 ──────────────────────────▶ 命名規則の策定・浸透
   │                                     │
課題抽出 ──────────────────────────▶ 承認フローの整備
   │                                     │
方針策定 ──────────────────────────▶ 運用ドキュメント整備
   │                                     │
移行ルール策定 ─────────────────────▶ 継続的な運用改善
   │
パイロット移行（本PJ完了）

※ 運用担当2名が既に常駐中のため、運用フェーズは協働で進める
```

## 11. スケジュール概要

| フェーズ | 期間 | 主な作業 |
|---|---|---|
| Phase 1: 要件定義・棚卸し | 2週間 | 現行環境調査、紛れ検出、要件確定 |
| Phase 2: 設計 | 2週間 | 移行設計、手順書作成、テストドメイン準備 |
| Phase 3: 環境構築・検証 | 2週間 | AD設定、EXO設定、テストドメインで検証 |
| Phase 4: パイロット移行 | 2週間 | パイロットドメイン切替、検証 |
| （計） | **8週間（2ヶ月）** | |

### Phase 1詳細

1. 現行環境の棚卸し
   - AD/Entra ID/EXO棚卸し
   - Linux環境（Postfix/Courier/DMZ SMTP）棚卸し
   - FireEye/GuardianWall設定確認
2. 紛れメールボックス検出・対処
3. **ID運用現状調査**
   - 現行のID作成・変更・削除フローのヒアリング
   - 命名規則、承認フローの確認
   - 課題の洗い出し
4. 要件確定、お客様承認

### Phase 2詳細

1. 移行設計書作成
2. ADスキーマ拡張
3. 手順書作成（運用、ユーザー向け）
4. テストドメイン準備
5. **ID運用方針策定**
   - EXO移行後のID運用方針の検討
   - 移行時のメールアドレス投入ルール策定

### Phase 3詳細

1. AD属性投入（テスト）
2. EXOコネクタ設定
3. GWC連携設定
4. テストドメインでのメール経路検証
5. フォールバック経路検証

### Phase 4詳細

1. お客様最終承認
2. AWS DMZ SMTP transport設定変更
3. パイロットドメインへのライセンス割当
4. 切替後の動作確認
5. 安定稼働確認

## 15. 用語定義

| 用語 | 定義 |
|---|---|
| EXO | Exchange Online |
| GWC | GuardianWall Cloud |
| Internal Relay | EXOの機能。メールボックスがない宛先を外部（指定コネクタ）に転送する |
| Accepted Domain | EXOで受け付けるメールドメイン |
| Authoritative | EXOがそのドメインの最終的な宛先（BOXがなければNDR） |
| proxyAddresses | AD属性。ユーザーのエイリアスメールアドレスを格納 |
| 紛れメールボックス | 意図せず作成されたEXO上の受信者オブジェクト |
| フォールバック経路 | EXOで受信者が見つからない場合の救済経路（内部DMZ SMTP→Courier IMAP） |
| GAL | グローバルアドレス一覧。組織内の連絡先一覧 |
| Mail User | EXOのオブジェクト。メールボックスは持たないが、アドレス帳に表示される |
| JML | Joiner-Mover-Leaver。IDライフサイクル管理の3フェーズ |
| Joiner | ID作成フェーズ（入社、新規作成） |
| Mover | ID変更フェーズ（異動、役職変更、アドレス追加） |
| Leaver | ID削除フェーズ（退職、アカウント廃止） |
